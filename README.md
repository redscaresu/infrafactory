# InfraFactory

[![CI](https://github.com/redscaresu/infrafactory/actions/workflows/ci.yml/badge.svg)](https://github.com/redscaresu/infrafactory/actions/workflows/ci.yml)
[![License: Apache 2.0](https://img.shields.io/badge/License-Apache_2.0-blue.svg)](LICENSE)
[![Go 1.24+](https://img.shields.io/badge/Go-1.24%2B-00ADD8?logo=go)](go.mod)

Scenario-driven OpenTofu generation and validation across **AWS**, **GCP**, and **Scaleway** тАФ generated by an LLM, validated against deterministic mock servers in seconds, optionally deployed against real cloud APIs.

## Why this exists

Hand-iterating IaC against real cloud APIs is slow, expensive, and flaky. LLMs are good at writing terraform but bad at debugging "why didn't this apply" тАФ the error messages are layers deep and the feedback loop is 90 seconds per attempt against a real cloud.

InfraFactory closes that loop. You write a scenario YAML declaring intent (resources + acceptance criteria). The pipeline generates HCL with an LLM, validates it through four layers (static тЖТ mock-deploy тЖТ real-deploy тЖТ destruction), and feeds structured failures back into the next iteration's prompt. Subsecond mock validation, no cloud credentials required.

## Quickstart

This walks through the GCP full-stack scenario тАФ VPC + GKE + Cloud SQL + GCS тАФ end-to-end against fakegcp. No GCP credentials, no real cloud calls.

```bash
# 1. Clone with siblings тАФ the three mock servers live in adjacent repos.
mkdir -p ~/dev && cd ~/dev
for repo in infrafactory mockway fakegcp fakeaws; do
  git clone https://github.com/redscaresu/$repo.git
done
cd infrafactory

# 2. Toolchain prerequisites
#    - Go 1.24+
#    - OpenTofu (https://opentofu.org) on PATH
#    - Claude CLI on PATH or OPENROUTER_API_KEY exported

# 3. Build infrafactory + start the three mocks side-by-side
make build           # produces ./bin/infrafactory
make mocks-up        # mockway:8080 (Scaleway) + fakegcp:8081 (GCP) + fakeaws:8082 (AWS)

# 4. Run the GCP full-stack scenario
./bin/infrafactory run scenarios/training/gcp-full-stack.yaml

# 5. Inspect generated HCL + per-iteration snapshots
ls output/gcp-full-stack/                       # final HCL the LLM converged on
ls .infrafactory/runs/gcp-full-stack/           # immutable per-run artifacts
```

A successful run ends with `Status: success` and `run/terminal_reason: pass (target_reached)`. If a validation layer fails, the failure JSON feeds into the next iteration's LLM prompt and the loop retries (default budget: 5 iterations).

To try the other clouds:
```bash
./bin/infrafactory run scenarios/training/aws-full-stack.yaml      # cloud: aws    тЖТ fakeaws
./bin/infrafactory run scenarios/training/full-stack-paris.yaml    # cloud: scaleway тЖТ mockway
```

## How it works

```
scenario YAML  тФАтФАтЦ╢  3-phase LLM generation  тФАтФАтЦ╢  4-layer validation  тФАтФАтЦ╢  retry on failure
   (intent)         plan тЖТ write HCL тЖТ review     static / mock / real / destroy   (5x budget)
```

**Three-phase generation** (`prompts/{aws,gcp,scaleway}/phase{1,2,3}*.md`):
1. **Plan architecture** тАФ scenario YAML + T-shirt size mappings тЖТ JSON architecture plan
2. **Generate HCL** тАФ architecture + cloud-specific pitfalls + provider schema тЖТ OpenTofu `.tf` files
3. **Self-review** тАФ generated HCL тЖТ 10-point checklist тЖТ corrections or `NO ISSUES FOUND`

**Four-layer validation** (each gates the next):
1. **Static** тАФ `tofu init/validate/plan` + OPA `deny` policies on the plan JSON
2. **Mock deploy** тАФ `tofu apply` against the matching mock; topology checks against `/mock/state`; OPA `deny_state` policies; mock-enforced FK integrity
3. **Real deploy** (optional, gated by `validation.layers.sandbox_deploy.enabled`) тАФ `tofu apply` against the real cloud with auto-destroy on failure
4. **Destroy verification** тАФ `tofu destroy` + orphan check against `/mock/state`

On failure, the structured failure (`layer`, `stage`, `check`, `detail`, `failure_class`) is appended to the next iteration's prompt as a `<feedback>` block so the LLM sees what specifically broke. Successful self-corrections can be promoted to per-cloud pitfall files for future runs (auto-learning, ADR-0012).

## Cloud coverage

Each cloud has the same set of extension points; the scenario's `cloud:` field drives every dispatch.

| Extension point | AWS | GCP | Scaleway |
|---|---|---|---|
| Mock server | [`fakeaws`](https://github.com/redscaresu/fakeaws) (`:8082`) | [`fakegcp`](https://github.com/redscaresu/fakegcp) (`:8081`) | [`mockway`](https://github.com/redscaresu/mockway) (`:8080`) |
| Provider pin | `hashicorp/aws ~> 5.70` | `hashicorp/google >= 5.0` (v5 for IAM SA) | `scaleway/scaleway >= 2.50` |
| Prompts | `prompts/aws/` | `prompts/gcp/` | `prompts/scaleway/` |
| Pitfalls | `pitfalls/aws.yaml` | `pitfalls/gcp.yaml` | `pitfalls/scaleway.yaml` |
| OPA policies | `policies/aws/` | `policies/gcp/` | `policies/scaleway/` |
| Training scenarios | `scenarios/training/aws-*.yaml` (11) | `scenarios/training/gcp-*.yaml` (8) | `scenarios/training/*-paris.yaml` (15) |
| Full-stack example | `aws-full-stack.yaml` | `gcp-full-stack.yaml` | `full-stack-paris.yaml` |

Each mock is wire-shape compatible with the matching real provider, enforced by an `examples/working/<svc>` smoke harness in the mock's own repo (`apply тЖТ plan -detailed-exitcode 0 тЖТ destroy`). See each mock's README for the API-compatibility contract тАФ it's the same shape across all three.

Adding a new cloud requires: prompt templates, pitfalls file, topology derivation rules, mock server, OPA policies, and training scenarios. Dispatch is driven by `cloudMockStateRouter`, `cloudConstraintPolicies`, `filterPolicyPathsByCloud`, `ExtractProviderSchemaForCloud`, and `detectCloud`.

## Commands

| Command | Purpose |
|---|---|
| `infrafactory init --path <file>` | Scaffold a new scenario YAML |
| `infrafactory validate <scenario>` | Layer 1 static checks only |
| `infrafactory generate <scenario>` | 3-phase LLM generation only |
| `infrafactory test <scenario>` | Layers 1-4 (no retry loop) |
| `infrafactory run <scenario>` | Full pipeline with retry loop + holdouts |
| `infrafactory mock start/stop/status/logs` | Manage the mockway container |
| `infrafactory ui` | Serve the web dashboard |

Key flags for `run`: `--clean` (fresh start), `--no-destroy` (keep resources for incremental follow-up), `--repair-iterations-max N` (retry budget, default 5).

## Acceptance criteria

Scenario YAML declares criteria that gate run success:

| Type | Layer 2 (mock) | Layer 3 (real) |
|---|---|---|
| `connectivity` | Topology graph query | TCP connect with retry |
| `http_probe` | Topology graph query | HTTP GET, expect 2xx/3xx |
| `dns_resolution` | Auto-pass (informational) | DNS A/AAAA lookup with retry |
| `policy` | OPA rules on plan + state | Same |
| `destruction` | Orphan check after destroy | Same + real destroy |

## Development

Pre-commit hook runs gitleaks + `make test` (Go unit + UI unit + Playwright e2e). Wire it once:
```bash
make install-hooks
```

Common targets:
```bash
make test           # full suite
make test-unit      # Go only
make ui-test-e2e    # Playwright only
make mocks-up       # start all three mocks
make mocks-down     # stop them
make mocks-status   # check
```

Gated e2e tests (cross-repo, require `tofu` + the sibling mock repos checked out):
```bash
INFRAFACTORY_ENABLE_E2E=1 go test ./internal/e2e/...
```
